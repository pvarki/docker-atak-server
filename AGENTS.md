# AGENTS.md — docker-rasenmaeher-takserver

## Purpose
TAK Server 5.x (Team Awareness Kit) packaged for the Deploy App (RASENMAEHER) ecosystem.
Provides military-grade blue-force tracking, situational awareness, and messaging for ATAK
(Android), WinTAK, and ITAK clients. Runs as a multi-container sidecar cluster sharing a
Docker network. This submodule wraps the upstream TAK Server distribution in a
RASENMAEHER-compatible compose and certificate setup.

## Stack & Key Technologies
- **Language:** Java (OpenJDK) — upstream TAK Server binary, not modified
- **TAK version:** 5.7-RELEASE-8 (configurable via `TAK_RELEASE` env var)
- **Database:** PostgreSQL `tak` schema
- **Certs:** JKS keystores generated by `takinit`; PKCS#12 compatible
- **Container pattern:** Multi-sidecar (6 containers share one Docker network: `taknet`)
- **Build tool:** Docker Compose + optional Gradle build for the distribution

## Development Setup
```bash
# Standalone TAK Server (for TAK-specific development):
cp takserver.env.example takserver.env
# Edit takserver.env: set TAK_RELEASE, TAKSERVER_CERT_PASS, TAK_CA_PASS

# Option A: Using gomplate (production-like)
docker compose pull --include-deps --ignore-pull-failures
docker compose -p tak up -d

# Option B: Local dev (rebuild containers, no gomplate)
export DOCKER_TAG_EXTRA="-dev"
docker build --no-cache --progress=plain \
  -t takserver:latest${DOCKER_TAG_EXTRA} .
docker compose -f docker-compose.local.yml -p tak up

# TAK volume must be deleted to pick up certificate changes:
docker compose -f docker-compose.local.yml -p tak down -v
docker compose -f docker-compose.local.yml -p tak rm -vf
```

## Code Conventions
- Environment variables: `TAKSERVER_*` and `TAK_*` prefixes

## Architecture Notes
**Multi-container sidecar pattern** — all containers share `taknet` network:

| Container    | Role                                          |
|--------------|-----------------------------------------------|
| takinit      | One-time cert + config generation; exits      |
| takconfig    | Main TAK Server (ports 8089-8094, 8443, 9000-9002) |
| takmsg       | Messaging broker (shares takconfig network)   |
| takapi       | REST API (shares takconfig network)           |
| takreten     | Retention cleanup (scheduled)                 |
| takplug      | Plugin manager                                |

**Ports exposed:**
- 8089: Standard CoT messaging (TCP/UDP)
- 8090–8094: Additional data feeds
- 8443: HTTPS REST API / Marti web UI
- 9000–9002: Federation

**Client package creation** (via REST API, preferred):
```bash
docker compose -p tak exec takserver_api /bin/bash -c \
  'CLIENT_CERT_NAME=replaceme /opt/scripts/make_client_zip.sh'
```

**Admin user creation:**
```bash
docker compose -p tak exec takserver_api /bin/bash -c \
  'cd /opt/tak/data/certs/ && CAPASS=$CA_PASS PASS=certpass \
   ./makeCert.sh client username && \
   ADMIN_CERT_NAME=username /opt/scripts/enable_admin.sh'
```

**OTA updates:** Place update files in `/opt/tak/webcontent/update` inside the container.

## Common Agent Pitfalls
1. **Certificate changes require a volume wipe.** If you change `TAKSERVER_CERT_PASS` or
   `TAK_CA_PASS`, you MUST delete the TAK volumes (`down -v`) before restarting. The old
   JKS files in the volume will conflict with new passwords.
2. **Never run `rmlocal` and `rmdev` simultaneously** — this is the general compose rule,
   but TAK is especially sensitive because it binds well-known ports (8089, 8443) that
   conflict immediately.
3. **TAK is a closed-source Java binary.** You cannot patch or modify the TAK Server
   application itself. All customization must be done via configuration files and scripts
   that mount into the container.
4. **`takintegration` shares the `taknet` network.** If TAK is misbehaving, check `takrmapi`
   logs as well — the integration layer mediates all user certificate provisioning.
5. **TAK cert files live in the volume, not the image.** Rebuilding the image does NOT
   regenerate certs. Only deleting the volume triggers cert regeneration via `takinit`.
